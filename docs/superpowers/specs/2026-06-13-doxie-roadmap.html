01

Where Doxie is today

A working proof of concept: for a single repo, author docs with Claude and publish them to Google Drive for non-engineers.

Shipped

  • doxie init — scaffolds templates, slash-command prompts, deploy stub.
  • doxie deploy — publishes doxie-docs/ to Drive; only changes push, docs update in place.
  • 4 slash commands — /create, /update, /ask, /deploy.
  • 3 high-altitude doc types — Overview, Feature, ADR (no code detail allowed).

The honest limits

  • One audience only — non-engineers, high altitude. Nothing an agent or on-call engineer can use to run or reason about the system.
  • Nothing watches the code — staying current is a manual chore.
  • It's an export, not a knowledge base — "markdown converted to Google Docs."

1Phase 1 — Trust

Believe it★ Flagship — build first

A doc is never silently wrong — for a human or an agent. The moment a code change makes a doc inaccurate, Doxie tells you and offers to fix it.

Kills P1·P2·P3  |  stakes are higher now: an agent will act on these docs — wrong context is worse than none.

New CLI command · the fast signal

doxie check

Reads your git changes and answers: did I break a doc, or leave something undocumented? Prints Drift + Gaps and exits non-zero. No AI key, milliseconds — hook-able into pre-push/CI later. Only nudges; never writes.

New slash command · the smart fix

/doxie:review

Takes a diff, judges which docs describe behavior the change altered, and drafts the edits across both tiers — respecting each type's rules (ADRs append-only). Confirms each before writing.

Code↔doc linkage — and why it's natural for the AI tier

v1 infers the link between code and doc. The upgrade lets a doc declare what it covers — especially natural for the technical tier, where a Runbook or System-Interactions doc maps almost one-to-one to specific code, commands, and config:

# in a doc, or a sidecar map
covers:
  - src/checkout/**
  - src/payments/stripe.ts
Decision

Linkage in doc front-matter (strip on deploy) vs. a sidecar .doxie/coverage.json.

→ Recommend the sidecar — keeps published human docs clean.

04

How we'll know it's working

Trust

Engineers run /doxie:review as part of shipping. "The doc was wrong" stops happening — for humans and for agents acting on the technical tier.

Depth & AI layer

An agent (or new hire) can run the service and understand its integrations straight from .doxie/context.md + the runbooks — without reverse-engineering the codebase.

Reach

A non-engineer finds the right doc themselves; an agent discovers other services' context maps via the catalog.

Intelligence

A cross-service question gets a correct, cited answer in seconds. Onboarding stops depending on any one person.

05

Open decisions

Forks worth settling before we build — none block starting Phase 1.

  1. Repo layout for the two tiers — how the technical tier sits next to the human tier, and how deploy knows which tier to publish (tier flag in the registry, or a publish-list in deploy.json).
  2. What generates .doxie/context.md — an init stub vs. a dedicated AI command (e.g. /doxie:map) vs. a mode of /create; and how drift detection keeps it current.
  3. Relationship to CLAUDE.md / AGENTS.md — is .doxie/context.md the canonical map those files point to, or does Doxie write into them directly?
  4. Linkage storage — covers: front-matter (strip on deploy) vs. sidecar .doxie/coverage.json. Recommend sidecar.
  5. /review vs /update — keep both, or fold update's drift step into review?
  6. Catalog & MCP substrate (Phases 3–4) — Drive manifest vs. index repo; how the MCP server is hosted. Defer until Phase 2 lands.
